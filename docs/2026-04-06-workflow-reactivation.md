# Workflow реактивации пользователей

Описание сценария, логики и всех состояний.

## 0. Контекст и точка входа

Пользователь прошёл flow получения сигналов, выбрал сегменты и нажал кнопку «Запустить кампанию». Система открывает экран создания кампании. Вход уже содержит готовый набор сигналов и сегменты — пользователю не нужно настраивать их повторно.

Три сегмента пользователей:

- **HL** (High Likelihood) — высокая вероятность возврата
- **L** (Likely) — средняя вероятность возврата
- **M** (Medium) — умеренная вероятность возврата

Пользователь взаимодействует с workflow через текстовые команды — он не настраивает ноды вручную, не заполняет формы. Система перестраивает граф в ответ на его инструкции.

## 1. Шаг 1 — базовый workflow (первая генерация)

Пользователь пишет: «Собери кампанию реактивации» — или это происходит автоматически при входе на экран. Система генерирует стартовый workflow без дополнительных настроек.

Схема: `signals → split → communication → check → branch → result`

| Блок | Описание |
|---|---|
| Signals | Входные данные: сигналы из предыдущего шага + уже выбранные сегменты HL / L / M. Пользователь не трогает этот блок. |
| Split | Маршрутизация по сегментам: каждый сегмент направляется в свою ветку коммуникации. |
| Communication | HL → Push (мягкий тон). L → Email. M → SMS. Первое касание с пользователем. |
| Check Response | Система ждёт события реакции. Условие: был ли отклик на коммуникацию. |
| Branch | YES (отклик есть) → Engaged flow. NO (нет отклика) → Retarget flow. |
| Engaged flow | Пользователь среагировал. Показываем баннер на сайте + дополнительный push. |
| Retarget flow | Пользователь не среагировал. Меняем канал, усиливаем оффер. |
| Result | Итог: reactivated (вернулся) или cold (не вернулся). |

## 2. Момент изменения — пользователь правит логику текстом

После генерации базового workflow пользователь начинает вносить правки через AI-команды. Каждая команда изменяет граф в реальном времени. Пользователь видит, как workflow перестраивается.

Команды пользователя:

- «Убери SMS, оставь только push и баннеры»
- «Добавь задержку 24 часа перед повторной коммуникацией»
- «Исключи пользователей, активных за последние 24 часа»
- «Если пользователь открыл email — не отправляй push»

## 3. Шаг 2 — обновлённый workflow (после всех изменений)

Схема: `signals → filter → split → communication → check → branch → engaged(с условием) / delay → retarget → second check → result`

### 3.1 Изменение 1 — убрать SMS

Команда: «Убери SMS, оставь только push и баннеры»

| Что изменилось | Описание |
|---|---|
| M → Push | Канал SMS удалён. Сегмент M теперь получает Push вместо SMS. |
| Retarget flow | В ветке retarget тоже убираем SMS. Повторная коммуникация — только push и баннеры. |

### 3.2 Изменение 2 — добавить Filter (activity 24h)

Команда: «Исключи пользователей, активных за последние 24 часа»

| Новый блок | Описание |
|---|---|
| Filter: Activity | Новый блок между Input и Split. Исключает пользователей, у которых было событие активности за последние 24 часа. Эти пользователи не попадают в workflow — они и так активны. |

### 3.3 Изменение 3 — добавить Delay 24h

Команда: «Добавь задержку 24 часа перед повторной коммуникацией»

| Новый блок | Описание |
|---|---|
| Delay 24h | Новый блок в ветке Retarget, перед повторной коммуникацией. Система ждёт 24 часа после первого касания, только потом запускает retarget. |
| Second Check | После retarget добавляется повторная проверка отклика. Итог: reactivated или cold. |

### 3.4 Изменение 4 — условие внутри Engaged flow

Команда: «Если пользователь открыл email — не отправляй push»

| Блок | Описание |
|---|---|
| Email открыт? (YES) | Пользователь уже проявил интерес через email. Отправляем только баннер на сайте. Push — излишнее давление, пропускаем. |
| Email открыт? (NO) | Email не открыт — пользователь ещё не среагировал на текст. Показываем баннер + отправляем push для усиления. |

## 4. Финальный workflow — полная схема всех блоков

Итоговый граф после всех четырёх изменений. Блоки отмечены статусом.

| Блок | Описание и статус |
|---|---|
| Signals + сегменты | Входные данные. HL / L / M. Без изменений. |
| Filter: Activity 24h | **НОВЫЙ.** Фильтр активности. Исключает недавно активных пользователей. |
| Split HL / L / M | Маршрутизация. Без изменений. |
| Push (HL) | Первый контакт: мягкий push. Без изменений. |
| Email (L) | Первый контакт: email. Без изменений. |
| Push (M) | **ИЗМЕНЕНО.** Ранее — SMS. Теперь push. |
| Check Response | Ожидание события реакции. Без изменений. |
| Branch YES/NO | YES → Engaged. NO → Delay → Retarget. Без изменений. |
| Email открыт? (YES) | **НОВОЕ** условие внутри Engaged. Только баннер, без push. |
| Email открыт? (NO) | **НОВОЕ** условие внутри Engaged. Баннер + push. |
| Delay 24h | **НОВЫЙ** блок в ветке Retarget. Ожидание 24 часов. |
| Retarget flow | **ИЗМЕНЕНО.** Push и баннеры, без SMS. |
| Second Check | **НОВЫЙ.** Повторная проверка после retarget. |
| Result | Reactivated / cold. Без изменений. |

## 5. Итоговое описание для агента

Этот документ описывает сценарий создания и итеративного редактирования workflow реактивации пользователей. Ключевые принципы для агента:

**Точка входа:** пользователь уже выбрал сегменты. Workflow генерируется автоматически при старте. Нет необходимости запрашивать сегменты или параметры — они переданы из предыдущего шага.

**Режим взаимодействия:** пользователь правит логику через текстовые команды. Агент должен интерпретировать инструкцию, определить, какой блок добавить / удалить / изменить, и перестроить граф. Не нужно переспрашивать или уточнять — достаточно явно показать, что изменилось.

**Структура изменений:**

- Удаление канала = заменить канал в соответствующей ветке сегмента на альтернативный
- Добавление блока = вставить новую ноду между существующими, сохранив связи
- Добавление условия = разбить существующую ноду на ветку с check-блоком
- Фильтр активности = всегда ставится перед Split, как первый блок обработки

**Цветовая кодировка (для визуальной схемы):**

- Серый — нейтральные блоки (вход, выход)
- Фиолетовый — маршрутизация (Split)
- Бирюзовый — каналы коммуникации
- Жёлтый / amber — условия и новые/изменённые блоки
- Коралловый — retarget flow
