# PRD: Peek-навигация шагов во FlowSignals

**Статус:** draft → ревью
**Автор:** Никита
**Тип:** доработка интерфейса
**Связанные сущности:** FlowSignals (поток создания сигнала), степпер

## TL;DR

В вертикальном скролле шагов FlowSignals соседние шаги (сверху и снизу от центрального) должны **выглядывать** из-за границ экрана как peek-карточки — чтобы юзер видел: «там есть ещё». Центральный шаг — это **focusedStep**, тот, который сейчас находится по центру вьюпорта. Он не обязан совпадать с activeStep (шагом, на котором юзер реально работает). Скролл свободный, степпер вверху позволяет прыгать к любому шагу — оба способа меняют focusedStep с magnetic snap к центру.

## Проблема

Сейчас шаги FlowSignals складываются в один длинный вертикальный скролл без визуальных границ контекста. Когда юзер на шаге N, он не видит:

- что есть шаг N−1 (можно вернуться назад)
- что есть шаг N+1 (что будет дальше)

Это убивает ощущение «где я в потоке» и мешает свободной навигации между шагами.

## Цель

Дать юзеру постоянное ощущение позиции в потоке и возможность свободно перемещаться вперёд-назад, не теряя визуального якоря.

## Ключевые понятия

| Термин | Значение |
|---|---|
| **activeStep** | Шаг, на котором юзер фактически работает (заполняет, настраивает). Меняется только по завершению предыдущего шага или явному переходу. |
| **focusedStep** | Шаг, который сейчас находится по центру вьюпорта. Меняется при скролле и при клике в степпере. |
| **peek** | Видимая часть карточки соседнего шага, торчащая из-за верхней или нижней границы вьюпорта. |

**Важно:** activeStep и focusedStep могут расходиться. Юзер может быть активен на шаге 3, но проскроллить вверх и рассматривать шаг 1 — focusedStep станет 1, activeStep останется 3.

## Механика

### Layout вьюпорта

```
┌─────────────────────────┐
│  степпер (1)─(2)─●─(4)  │   ← всегда виден, показывает activeStep
├─────────────────────────┤
│ ▓▓▓ peek шага N−1 ▓▓▓   │   ← видна верхушка карточки
│                         │
│   ┌─────────────────┐   │
│   │                 │   │
│   │  focusedStep N  │   │   ← по центру, в фокусе
│   │                 │   │
│   └─────────────────┘   │
│                         │
│ ▓▓▓ peek шага N+1 ▓▓▓   │   ← видна нижушка карточки
└─────────────────────────┘
```

### Поведение peek

- Высота peek соседнего шага: ~10–15% высоты карточки (визуально достаточно, чтобы прочесть заголовок/иконку, но не отвлекает).
- Если шага сверху нет (focusedStep — первый) → peek сверху отсутствует, верхний край пустой.
- Если шага снизу нет (focusedStep — последний) → peek снизу отсутствует.
- Peek-карточки приглушены (opacity ~60% или fade-out градиент к краю экрана) — чтобы не конкурировать с focusedStep.

### Поведение скролла

- Скролл **свободный**: юзер может прокручивать колесом/тачем в любом направлении.
- При остановке скролла срабатывает **magnetic snap**: ближайший шаг центрируется автоматически и становится focusedStep.
- Snap — мягкий, ~150–200 ms ease-out, без рывков.
- Между шагами есть «мёртвая зона»: пока юзер скроллит, центрирование не дёргается; snap включается только после паузы.

### Поведение степпера

- Степпер вверху (горизонтальная цепочка точек/нумерованных пунктов) показывает **activeStep** (визуально выделен) и доступные для перехода шаги.
- Клик по любому пройденному шагу в степпере = плавный скролл к нему, focusedStep становится этим шагом. activeStep при этом **не меняется**.
- Шаги «вперёд» от activeStep в степпере недоступны (если они ещё не разблокированы).

### Поведение activeStep

- activeStep двигается только при явном завершении текущего шага (например, кнопкой «Далее»).
- Если юзер ушёл скроллом/степпером назад на шаг 2, а activeStep = 4 — степпер продолжает показывать «вы работаете над шагом 4», а focusedStep = 2 показывает контент шага 2 в режиме просмотра/редактирования.
- При завершении activeStep автоматически: focusedStep синхронизируется с новым activeStep (плавный скролл к нему).

## Состояния

| Сценарий | focusedStep | activeStep | Peek сверху | Peek снизу |
|---|---|---|---|---|
| Юзер только зашёл, шаг 1 | 1 | 1 | — | шаг 2 |
| Юзер на середине потока, шаг 3 | 3 | 3 | шаг 2 | шаг 4 |
| Юзер на activeStep 4, проскроллил наверх | 1 | 4 | — | шаг 2 |
| Юзер завершил последний шаг | N | N | шаг N−1 | — |

## Edge cases

1. **Только один шаг в потоке** → peek нет ни сверху, ни снизу. Карточка просто по центру.
2. **Шаг выше высоты вьюпорта** → focusedStep скроллится **внутри своей карточки** (внутренний скролл), peek соседних шагов остаются по краям. Magnetic snap не срабатывает, пока юзер не доскроллит до края карточки.
3. **Очень быстрый скролл (flick)** → snap срабатывает на тот шаг, на котором скролл остановился, без «перелёта».
4. **Юзер начал заполнять focusedStep ≠ activeStep** → если шаг разрешает редактирование (например, пройденный) — изменения сохраняются и activeStep остаётся прежним. Если шаг недоступен (вперёд от activeStep) — карточка в read-only/disabled состоянии.
5. **Клавиатурная навигация** (Tab, стрелки) → должна синхронизироваться с focusedStep, чтобы фокус всегда был виден.

## Открытые вопросы

1. **Тайминг snap:** 150 мс — предположение, нужно покрутить в прототипе.
2. **Высота peek:** 10–15% — тоже предположение. Возможно, для коротких/высоких шагов нужна разная логика.
3. **Что если activeStep вне вьюпорта надолго?** Нужен ли «pill» внизу/сверху типа «↓ вернуться к шагу 4», чтобы быстро прыгнуть обратно на activeStep?
4. **Скролл внутри длинной карточки** vs **скролл между шагами** — нужно явно протестировать, чтобы не было конфликта жестов.

## Что не входит в скоуп

- Анимации перехода между шагами (отдельная история).
- Изменения в самих карточках шагов (контент, кнопки).
- Логика разблокировки шагов вперёд от activeStep.
